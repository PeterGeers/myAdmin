## 1. Test vs Production — Feature Does Not Exist in the App

**Finding**: The standalone page `getting-started/test-vs-production/` describes a UI toggle that does not exist. The `first-login/#4-test-or-production-mode` section references the same non-existent toggle.

**Assessment**: Confirmed after code review. There is **no test/production toggle in the frontend UI**. The test mode is controlled by a hardcoded `flag = False` variable in `backend/src/app.py` (line 115). There is no API endpoint to switch modes and no frontend component for it. This is a developer/deployment-level setting, not a user-facing feature.

The documentation describes a feature ("click the toggle in the top-right corner to switch modes") that users will never find in the application. This is misleading.

**Recommendations**:

- **Option A — Remove both pages entirely**: Since the feature doesn't exist for end users, remove `test-vs-production.md` / `.en.md` and the "4. Test or Production mode" section from `first-login.md` / `.en.md`. Also remove the nav entry from `mkdocs.yml`. This is the cleanest approach.
- **Option B — Rewrite as a system admin topic**: If test/production mode is relevant for SysAdmins who deploy the system, move the content to the `admin/` section and rewrite it to explain that mode is set via the `flag` variable in `app.py` or environment configuration. Remove all references to a UI toggle.
- **Either way**: Remove the "4. Test or Production mode" section from `first-login.en.md` and `first-login.md` — it references something users cannot do.
- **Update the help links**: Check if `helpLinks.ts` maps anything to the test-vs-production page and remove it if so.

Decision: Option A

## 4. Parameters — Clarify the Two-Level System

**Finding**: Add documentation about managing parameters, listing what they do, with extra focus on storage parameters.

**Assessment**: The previous feedback described the Advanced tab incorrectly as the main place for parameter management. After reviewing the actual code, the parameter system has two distinct levels that serve different audiences:

**How parameters actually work — scope inheritance chain:**

The system resolves parameters by walking: `user → role → tenant → system`. The first value found wins. This means:

- **System-scope parameters** are defaults defined in `parameter_schema.py` and seeded from `module_registry.py`. They provide baseline values for all tenants and only change when the software changes. SysAdmins can view and edit these via the **Advanced tab** (🔧), which shows the raw `ParameterManagement` table.
- **Tenant-scope parameters** are overrides that tenant administrators set through the **structured UI tabs** (Storage, Financial, etc.). When a tenant admin configures their storage provider, sets their fiscal year, or enters branding details, those are written as tenant-scope parameters that override the system defaults.

**What the tenant admin sees and can configure (through structured tabs):**

| Tab             | Parameters managed | Examples                                                                                |
| --------------- | ------------------ | --------------------------------------------------------------------------------------- |
| **Storage**     | `storage.*`        | Storage provider (gdrive/s3_shared/s3_tenant), Google Drive folder IDs, S3 bucket names |
| **Financial**   | `fin.*`            | Default currency, fiscal year start month, locale                                       |
| **Tenant Info** | Company details    | Company name, address, bank details                                                     |

**What the SysAdmin sees (Advanced tab — raw parameter table):**

The Advanced tab shows ALL parameters across all namespaces in a flat key-value table. SysAdmin can create, edit, and delete at both system and tenant scope. This is the "escape hatch" for configuration that doesn't have a structured UI yet.

**The parameter schema** (`parameter_schema.py`) defines the full set of known parameters with metadata (labels in NL/EN, types, defaults, dropdown options, conditional visibility). Current schema sections:

| Namespace      | Label              | Module | Key parameters                                                                   |
| -------------- | ------------------ | ------ | -------------------------------------------------------------------------------- |
| `storage`      | Storage Settings   | —      | `invoice_provider`, Google Drive folder IDs, S3 bucket names, report output path |
| `str_branding` | STR Branding       | STR    | Company logo, name, address, VAT, CoC, email                                     |
| `zzp_branding` | ZZP Branding       | ZZP    | Company logo, name, address, VAT, CoC, phone, email                              |
| `fin`          | Financial Settings | FIN    | Default currency, fiscal year start month, locale                                |
| `str`          | STR Settings       | STR    | Number of rooms, beds, active platforms                                          |

Additionally, `module_registry.py` defines per-module parameters (ZZP invoice prefix, payment terms, debtor/creditor accounts, field configs for contacts/products/invoices/time entries) that are seeded when a module is activated.

**Recommendations for documentation:**

1. **Don't document the Advanced tab for end users** — it's SysAdmin-only and shows raw data. Instead, document the structured tabs that tenant admins actually use.

2. **Expand the Storage tab documentation** with:
   - The three provider options and what each means
   - For Google Drive: explain the folder IDs (root, invoices, templates) and how to find them
   - For S3: explain shared vs tenant bucket
   - The credential upload / OAuth flow
   - The "Test Connection" button

3. **Add a "Parameters" section for tenant admins** explaining:
   - Parameters control how the system behaves for your organization
   - They are pre-configured with sensible defaults when modules are activated
   - You can customize them through the structured settings tabs (Storage, Financial, etc.)
   - Some parameters (like branding) affect how invoices and documents look
   - Some parameters (like field configs) control which fields are visible/required in forms

4. **For SysAdmin documentation** (in the admin section, not tenant-admin):
   - Document the Advanced tab as a raw parameter editor
   - Explain the scope inheritance (system → tenant → role → user)
   - Note that system-scope parameters are defaults that apply to all tenants unless overridden

### Deployment Compatibility — Local Docker vs Railway

Adding screenshots is fully compatible with both environments, but the current deployment setup requires an extra step to keep things in sync:

**How docs are served today:**

| Environment      | Docs source                                                | How it gets there                                                              |
| ---------------- | ---------------------------------------------------------- | ------------------------------------------------------------------------------ |
| **Local Docker** | `docs/site/` → mounted as `/app/docs-site`                 | Volume mount in `docker-compose.yml`: `./docs/site:/app/docs-site`             |
| **Railway**      | `backend/docs-site/` → baked into the image                | Pre-built and committed to git (Railway builds from `./backend/` context only) |
| **Local dev**    | Falls back to `../../docs/site` relative to `backend/src/` | Automatic fallback in `app.py`                                                 |

**What this means for screenshots:**

1. Images go in the MkDocs source: `docs/docs/assets/images/` (subfolders per section)
2. Run `mkdocs build` from the `docs/` folder — this outputs to `docs/site/`, which includes all images in `docs/site/assets/images/`
3. **For Railway**: the built `docs/site/` must be copied to `backend/docs-site/` before committing. This is the same process used today for any docs change — screenshots don't introduce a new step, they just add files to the existing flow.
4. **For local Docker**: the volume mount picks up `docs/site/` automatically after `mkdocs build`. No extra step needed.

**Recommendation**: Consider adding a simple build script (e.g., `scripts/build-docs.sh`) that runs `mkdocs build` and copies the output to `backend/docs-site/` in one step. This would prevent forgetting the copy step and applies to all docs changes, not just screenshots.

Decision: "No action at this moment. There is a hook in the git flow that compiles and uploads it to the right spot"

**Image size considerations**: Keep screenshots optimized (PNG compressed or WebP). Large images increase the git repo size since `backend/docs-site/` is committed. A reasonable target is under 100KB per screenshot. Tools like TinyPNG or `pngquant` can help.
